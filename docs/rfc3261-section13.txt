13  Initiating a Session

13.1  Overview

   The INVITE method is used to initiate a session between two user
   agents. The UAC (caller) sends an INVITE; the UAS (callee) responds
   with provisional responses (1xx) and ultimately a final response.

   The INVITE/200/ACK three-way handshake is the core of session setup:

     UAC ---INVITE--->  UAS
     UAC <--100 Trying- UAS   (optional, sent immediately)
     UAC <--180 Ringing UAS   (optional, UAS is alerting user)
     UAC <--200 OK----  UAS   (user answered)
     UAC ---ACK------->  UAS  (confirms receipt of 200)
     <== media flows ==>

   Note: ACK for a 2xx response is end-to-end (not handled by the
   transaction layer). ACK for non-2xx responses IS handled by the
   transaction layer.

   The INVITE establishes a dialog. The dialog is identified by the
   combination of Call-ID, From tag, and To tag.

13.2  UAC Processing

13.2.1  Creating the Initial INVITE

   The UAC MUST include the following header fields in the INVITE:
     - Request-URI: address of the callee
     - To: logical recipient (address-of-record)
     - From: identity of the caller, with a tag parameter
     - Call-ID: globally unique identifier for this call
     - CSeq: sequence number + method name ("INVITE")
     - Max-Forwards: SHOULD be 70
     - Via: transport + branch parameter (z9hG4bK...)
     - Contact: URI at which the UAC wishes to receive requests

   The INVITE SHOULD include:
     - An Allow header listing supported methods
     - A Supported header listing supported extensions
     - A body (typically SDP) describing the offered media

   SDP offer/answer model: the UAC puts its SDP offer in the INVITE
   body. The UAS puts its SDP answer in the 200 OK body. This two-phase
   exchange negotiates codecs, ports, and media types.

   The UAC SHOULD NOT send another INVITE to the same destination
   until it receives a final response to the first one (or the
   transaction times out).

13.2.2  Processing INVITE Responses

13.2.2.1  1xx Responses

   1xx responses are provisional. They do not terminate the transaction.
   The UAC MUST be prepared to receive multiple 1xx responses.

   - 100 Trying: sent immediately by a proxy or UAS to stop
     retransmissions. Does not establish a dialog.
   - 180 Ringing: the UAS is alerting the user (phone is ringing).
     This establishes an early dialog.
   - 183 Session Progress: used to send early media or ringback tones.

   For each 1xx with a To tag that is different from previous responses,
   the UAC constructs a new early dialog.

13.2.2.2  3xx Responses

   The UAC SHOULD try the URIs in the Contact header of the 3xx
   response, following the redirect. A new INVITE is generated for each
   redirect target.

13.2.2.3  4xx, 5xx and 6xx Responses

   These are failure responses and terminate the INVITE transaction.
   The UAC MUST send an ACK to any final response, including failures.
   (ACK for non-2xx is generated by the client transaction layer.)

   Notable 4xx codes for INVITE:
     - 400 Bad Request
     - 401 Unauthorized (auth challenge)
     - 403 Forbidden
     - 404 Not Found
     - 480 Temporarily Unavailable
     - 486 Busy Here
     - 487 Request Terminated (response to CANCEL)
     - 488 Not Acceptable Here (SDP offer rejected)

   A 6xx response (e.g. 600 Busy Everywhere) is a global failure —
   the UAC MUST NOT retry at an alternate location.

13.2.2.4  2xx Responses

   A 200 OK to an INVITE terminates the transaction and establishes
   the dialog. The UAC MUST:
     1. Send an ACK immediately (this ACK is a separate transaction,
        end-to-end, NOT retransmitted by the transaction layer)
     2. Extract the SDP answer from the 200 OK body
     3. Store the Contact URI from the 200 OK for future in-dialog
        requests (BYE, re-INVITE, etc.)
     4. Store the To tag for dialog identification

   The ACK is constructed as follows:
     - Request-URI: Contact URI from the 200 OK
     - To: same as INVITE To, but WITH the To tag from the 200 OK
     - From: same as INVITE From (with From tag)
     - Call-ID: same as INVITE
     - CSeq: same number as INVITE, but method is "ACK"
     - Via: NEW branch parameter (ACK is a new transaction for non-2xx;
       for 2xx it is technically outside the original transaction)

   If multiple 2xx responses arrive (due to forking), the UAC MUST
   send an ACK for each one. It SHOULD then send a BYE to terminate
   the dialogs it does not wish to keep.

13.3  UAS Processing

13.3.1  Processing of the INVITE

   Upon receiving an INVITE, the UAS MUST:
     1. Check for merged requests (same From tag + Call-ID + CSeq
        arriving via different paths due to forking) — respond 482
        Loop Detected to duplicates.
     2. Inspect the Require header. If an unsupported option tag is
        listed, respond 420 Bad Extension with an Unsupported header.
     3. Process the SDP offer in the body (if present).
     4. Decide whether to accept, redirect, or reject the INVITE.

13.3.1.1  Progress

   While the UAS is alerting the user or processing the request, it
   SHOULD send provisional responses to stop UAC retransmissions:

     - 100 Trying: SHOULD be sent immediately if not sent by a proxy.
     - 180 Ringing: SHOULD be sent when the UAS is alerting the user.

   Provisional responses establish early dialogs. The UAS MUST add a
   To tag to all responses (except 100 Trying, where it is optional).

   The UAS may send early media using 183 Session Progress with an SDP
   answer body before sending the final 200 OK.

13.3.1.2  The INVITE is Redirected

   The UAS MAY redirect the INVITE to one or more alternative locations
   by returning a 3xx response with Contact header values. The dialog
   is not established; the UAC retries at the redirected URI(s).

13.3.1.3  The INVITE is Rejected

   If the UAS decides not to accept the INVITE (user rejects, busy,
   SDP incompatibility, etc.), it sends a failure response:

     - 486 Busy Here: user is currently busy
     - 488 Not Acceptable Here: SDP offer not acceptable (should
       include a Warning header explaining why)
     - 403 Forbidden: UAS refuses without explanation
     - 480 Temporarily Unavailable

   The UAS MUST NOT send a BYE in response to a rejected INVITE;
   the error response itself terminates the transaction.

13.3.1.4  The INVITE is Accepted

   When the user answers (or the UAS auto-accepts), the UAS sends
   a 200 OK. The 200 OK MUST include:
     - A Contact header with a URI at which the UAS can be reached
       for subsequent in-dialog requests
     - A To header with a To tag (establishes the dialog)
     - An SDP answer body (accepting/modifying the UAC's offer)
     - Allow header (RECOMMENDED)

   After sending the 200 OK, the UAS starts a timer. If no ACK is
   received, the UAS retransmits the 200 OK (unlike other responses,
   the 200 OK to INVITE is retransmitted by the TU, not the transaction
   layer, because ACK is end-to-end). If no ACK arrives after 64*T1
   seconds, the UAS SHOULD send a BYE to terminate the dialog.

   When the ACK arrives, the session is established. Media can flow.

================================================================================

Key points for siprectify implementation:

1. As a SIPREC Recording Server (SRS), siprectify acts as a UAS.
   It receives INVITE (from the SRC/SBC), sends 100 Trying + 180 Ringing
   (optional) + 200 OK, then waits for ACK before starting to receive RTP.

2. The 200 OK response MUST contain:
   - SDP answer advertising the IP:port pairs where siprectify will
     receive RTP (one port per stream/leg for SIPREC dual-stream)
   - Contact header
   - To tag (generated by siprectify)

3. ACK handling: siprectify must handle the ACK to know when to start
   expecting RTP packets. Before ACK, the ports are open but no media
   should arrive.

4. BYE handling: when the call ends, the SRC sends BYE. siprectify
   responds 200 OK and stops recording.

5. SIPREC-specific: the INVITE will have Require: siprec and a multipart
   body (SDP + rs-metadata XML). The SDP will have two m= lines (one per
   participant), each with a=label: attribute for stream identification.
   siprectify must parse both the SDP AND the metadata XML to know which
   stream is caller vs callee.
